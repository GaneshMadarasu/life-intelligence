Section A

What is this?

Your life is already full of data — lab results, prescription bottles, gym records, insurance policies, employment contracts — but it's all scattered across disconnected apps, filing cabinets, and PDF folders. This system reads all of it and builds a single connected map of your life.

💡 Think of it like a brilliant personal assistant who has read every document you've ever owned, remembers every connection between them, and can instantly answer questions like: "Given my diabetes and my health insurance, am I maximizing my FSA savings account?" — a question no single app could answer alone.

The system uses two types of search at the same time: semantic search (finds text that means the same thing even in different words) and graph traversal (follows the connections — this medication → treats this condition → covered by this insurance plan). The results are combined and handed to Claude, which writes a precise answer grounded entirely in your own data.

Everything runs on your laptop. Nothing is sent to the cloud except the question text and relevant excerpts you choose to ask about. Neo4j and ChromaDB run as local Docker containers. The API listens only on 127.0.0.1.

Section C

What happens when you ask a question?

A step-by-step story — from your question to your answer.

1

📝 You ask a question

You type in the web chat UI at localhost:8000, the CLI, or call POST /query directly. The UI maintains a rolling conversation history (last 5 exchanges) so follow-up questions like "and how does that compare to last week?" work naturally.

Web UI: http://127.0.0.1:8000 CLI: python scripts/query.py API: POST /query + conversation_history[ ]
2

🔍 Two searches fire in parallel + Whoop injection

The Graph Retriever runs Cypher queries against Neo4j — traversing relationships like "Person → TAKES_MEDICATION → Medication → INTERACTS_WITH → Supplement". The Vector Retriever finds semantically similar text chunks in ChromaDB. If the question mentions fitness keywords (HRV, recovery, strain, sleep, workout), the graph retriever also fetches the last 14 days of live Whoop biometrics directly from WhoopRecovery, WhoopCycle, SleepRecord, and Workout nodes.

Graph: Cypher traversal in Neo4j Vector: Semantic similarity in ChromaDB Whoop: live biometric injection (keyword-triggered)
3

⚖️ Results are fused and ranked

The Hybrid Retriever merges both result sets. Graph results get 60% of the weight, vector results 40%. Recent events (last 5 years) get a small recency boost. Duplicates are removed. The top K results become the "context" for the AI.

Graph weight: 60% Vector weight: 40% Recency boost: up to +15%
4

🛡️ Safety checks run automatically

Before generating the answer, the Safety Checker scans for: known drug-drug interactions (e.g. Sertraline + Escitalopram — serotonin syndrome risk), supplement-drug conflicts (e.g. Berberine + Metformin — hypoglycemia risk), insurance gaps, and overdue genetic screenings. High-severity warnings are prepended to the AI's context.

HIGH: Drug interactions MEDIUM: Supplement conflicts LOW: Insurance gaps
5

🔗 Cross-domain links are surfaced

The Cross-Domain Linker checks whether any connections exist across domains — for example, an insurance plan (finances) that covers a condition (healthcare), or an employment contract benefit (legal) that funds medication costs. These are included as context for the AI.

InsurancePlan ↔ Condition Contract ↔ Benefit Expense ↔ Procedure
6

🤖 Claude reads your data and writes the answer

Claude claude-sonnet-4-6 receives: your conversation history (prior Q&A turns) + the current question with freshly retrieved context + any safety warnings. Prior turns are passed as actual Anthropic message turns — so the model naturally refers back to earlier questions. It is instructed to only use the provided context, cite specific values (dates, dosages, biometric readings), and lead with safety warnings if any exist.

Model: claude-sonnet-4-6 Context: ~6,000 chars of your data History: last 5 exchanges (10 messages) Grounded: no hallucination outside context
7

✅ You get a structured answer

The response includes: the answer text, key facts as bullet points, cross-domain insights discovered, safety warnings sorted by severity, source files cited, and a confidence level (high if lots of context found, medium if sparse).

answer key_facts[ ] cross_domain_insights[ ] warnings[ ] sources[ ] confidence

Section E

Whoop Integration — Live Biometric Pipeline

How data flows from your Whoop device into the knowledge graph and into AI answers.

1

⌚ OAuth2 Authentication

Run python scripts/whoop_sync.py auth. The client starts a local HTTP server on port 8080, opens your browser to the Whoop authorization page, and captures the callback code via a thread-safe queue. Tokens are saved to .whoop_tokens.json (chmod 600). If the automatic callback fails, use --code to paste the redirect URL manually.

Local callback server on :8080 queue.Queue for thread-safe code capture Auto token refresh (60s buffer before expiry)
2

📡 API Sync (Mixed Versions)

The Whoop API uses two base URLs: /developer/v1 for daily cycles, and /developer/v2 for recovery, sleep (/activity/sleep), and workouts (/activity/workout). The sync engine paginates all endpoints using nextToken cursors and writes results into the mapper.

v1: /cycle v2: /recovery v2: /activity/sleep v2: /activity/workout
3

🗺️ Mapping → Graph Nodes

The mapper translates Whoop API shapes into the internal schema. Each sync writes idempotent MERGE statements — re-syncing the same day never creates duplicates.

WhoopRecovery: recovery_score, hrv_rmssd, resting_hr, spo2_pct WhoopCycle: strain, calories, avg/max HR SleepRecord (source=whoop): duration, deep/REM, performance %, efficiency % Workout (source=whoop): type, strain_score, HR zones 1–5
4

🧠 Context Injection at Query Time

The graph retriever detects fitness keywords in the query (HRV, recovery, strain, sleep, workout, etc.) and fetches the last 14 days of Whoop nodes, formatting them as text lines prefixed with === Whoop Biometric Data ===. The hybrid retriever prepends these verbatim to the AI context, ahead of other results, ensuring Claude always sees live biometric data for fitness questions.

Keyword-triggered (no extra API call) 14-day window per query Prepended ahead of document/entity results

Section G

Safety System — Automatic Warnings

Every query runs a safety check before the AI answers. Warnings are injected into the AI's context so they always surface — even if you didn't ask.

🔴 HIGH — Drug Interactions

Flags medications that interact dangerously. Seeded example: Sertraline + Escitalopram → duplicate serotonin mechanism → serotonin syndrome risk.

🔴 HIGH — Supplement Conflicts

Flags supplements that conflict with medications. Seeded example: Berberine + Metformin → both lower blood glucose → additive hypoglycemia risk.

🔴 HIGH — Pharmacogenomics

Flags when your genetic metabolism type affects a current drug. Seeded: CYP2C19 poor metabolizer + Escitalopram → elevated plasma levels → dose review needed.

🟡 MEDIUM — Insurance Gaps

Flags active chronic conditions (e.g. Type 2 Diabetes) with no linked insurance plan coverage — surfaced once the finances domain is active.

🟡 MEDIUM — Upcoming Obligations

Flags contract obligations due in the next 30 days — surfaced once the legal-contracts domain is active.

🟢 LOW — Overdue Screenings

Flags high genetic risks (e.g. BRCA2) with no recent related procedure in the past year. Prompts action before a problem develops.

Section I

Why was it built this way?

Key architectural choices explained as cause-and-effect.

Graph database (Neo4j) instead of a regular database

Medical knowledge is fundamentally a network: Drug A interacts with Drug B, which metabolized by Gene C, which increases Risk D, which is covered by Insurance Plan E. A regular database would need 5 JOINs to answer a single question. A graph database follows those connections in milliseconds, in any direction.

Umbrella architecture with pluggable domains

Instead of one monolithic healthcare app, every life domain is self-contained and interchangeable. Adding finances later requires zero changes to the core — just a new src/domains/finances/ folder extending BaseDomain. The backbone schema (Person, Document, TimePoint) never changes, no matter how many domains are added.

Hybrid retrieval (graph 60% + vector 40%) instead of just one

Graph search is precise but brittle — it finds exactly what matches the relationship, but misses paraphrases. Vector search is broad but imprecise — it finds semantically similar text, but loses structured facts. Combining them at a 60/40 ratio gives the precision of graph traversal with the coverage of semantic similarity.

Claude extracts entities from documents (not hand-written parsers)

Medical documents are wildly inconsistent: different hospitals format lab results differently, medications use brand or generic names interchangeably, dates appear in 10 different formats. Writing a regex parser for all of that would take months and break constantly. Claude can read any format and return clean structured JSON reliably.

Safety warnings are injected automatically — never optional

Drug interactions are life-critical. A user asking "what's my HbA1c trend?" should still be told about the Sertraline + Escitalopram risk if it exists, even though they didn't ask about it. The Safety Checker runs on every query and prepends HIGH-severity warnings to the AI's context before it writes the answer.

Everything runs locally — API bound to 127.0.0.1 only

Personal health records, genetic data, and financial history are the most sensitive data a person owns. Neo4j and ChromaDB run as local Docker containers. The FastAPI server listens only on localhost — it cannot be reached from any other machine on the network. The only data that leaves the machine is the question text + excerpts sent to the Claude API.

Whoop biometrics injected at query time, not at ingest time

Whoop data is already in Neo4j after each sync. Rather than re-embedding it into ChromaDB (which would make vector search noisy with numeric time-series data), the graph retriever directly fetches the last 14 days of biometric nodes when fitness keywords appear in the query. This keeps vector search for semantics and graph traversal for structured data — each doing what it does best.

Conversation history passed as real message turns, not stuffed into the prompt

A naive approach would concatenate prior Q&A into the system prompt. Instead, prior turns are passed as actual {"role": "user"} / {"role": "assistant"} messages in the Anthropic messages array. This lets the model use its native multi-turn reasoning — it knows the difference between what it said and what you said, and can refer back naturally. History is capped at 10 messages to keep token usage bounded.

Two-layer caching: semantic TTL cache + Anthropic prompt caching

A SHA-256 in-memory semantic cache with configurable TTL (default 5 min) avoids calling Claude for identical questions. Conversation-history queries bypass it — they're context-dependent by definition. Simultaneously, Anthropic prompt caching (cache_control: ephemeral on the system block + anthropic-beta: prompt-caching-2024-07-31 header) caches the ~2 KB system prompt across calls, saving ~80% of system-prompt tokens on cache hits.

Async-safe Neo4j: blocking driver wrapped in run_in_executor

The official Neo4j Python driver is synchronous. FastAPI's async request handlers would block the event loop if Neo4j queries ran directly. Every Neo4j call is wrapped with asyncio.get_event_loop().run_in_executor(None, lambda: …) via a _run_sync() helper. This keeps the API fully async without replacing the Neo4j driver or using experimental async alternatives.

SSE streaming (POST /query/stream) uses the Anthropic messages.stream() context manager

The streaming endpoint uses client.messages.stream() as a context manager, iterating text_stream and yielding data: <chunk>\n\n SSE events via FastAPI's StreamingResponse. The UI consumes it with response.body.getReader() + TextDecoder, appending tokens character-by-character with a blinking cursor. Streaming bypasses the semantic cache (since partial answers shouldn't be cached) but still benefits from Anthropic prompt caching.

Glossary — Words You Might Not Know

Every technical term used in this project, explained in plain English.

GraphRAG
Retrieval-Augmented Generation using a knowledge graph. Instead of just searching documents, you traverse a network of connected facts before generating an answer.
Vector Embedding
A list of ~300 numbers that represents the meaning of a sentence. Two sentences about the same topic will have similar numbers, even if they use different words.
Graph Database (Neo4j)
A database that stores connections between things, not just rows and columns. Like a mind-map you can query: "find everything connected to Metformin within 2 hops."
Cypher
Neo4j's query language. Like SQL but for graphs. MATCH (p:Person)-[:TAKES_MEDICATION]->(m:Medication) means "find all medications this person takes."
ChromaDB
A local database that stores text embeddings (vectors) and lets you find the most similar chunks to a query. Runs entirely on your machine — no cloud needed.
Vertical
A sub-category within a domain. The healthcare domain has 4 verticals: medical, fitness, mental health, genetics. Each vertical has its own node types and queries.
MERGE (Cypher)
A Neo4j command that means "create this node if it doesn't exist, otherwise update it." This makes ingestion safe to repeat — no duplicates ever created.
Pharmacogenomics
The study of how your genes affect how your body processes medications. For example, if you're a "CYP2C19 poor metabolizer," certain antidepressants build up to higher levels in your blood.
HbA1c
A blood test that shows your average blood sugar level over the past 3 months. Used to monitor diabetes management. Normal is below 5.7%; diabetic is 6.5% or above.
Recency Boost
A small score bonus given to recent events during retrieval. A lab result from last month is more relevant than one from 10 years ago — the system accounts for this.
Hybrid Retrieval
Using two search methods at the same time — graph traversal (precise, structured) and vector similarity (broad, semantic) — then blending the results into one ranked list.
BaseDomain / BaseVertical
Abstract Python classes that define the contract every domain and vertical must follow. Like a job description — any new domain just needs to fill in the blanks and it plugs in automatically.
HRV (Heart Rate Variability)
The variation in time between consecutive heartbeats, measured in milliseconds. Higher HRV generally indicates better recovery and nervous system health. Whoop reports this as hrv_rmssd.
Strain (Whoop)
A 0–21 scale measuring cardiovascular load for the day or a workout. A strain of 14+ is high exertion; below 10 is light activity. Stored in WhoopCycle.strain and Workout.strain_score.
Recovery Score (Whoop)
A 0–100% daily readiness score based on HRV, resting heart rate, sleep performance, and SpO2. Green (67–100%) means ready to train; yellow (34–66%) means moderate; red (<34%) means rest.
Multi-turn Conversation
A conversation where the AI remembers what was said earlier. Enabled by passing prior Q&A as message history to the Claude API, so follow-up questions like "and how does that compare to last week?" work without repeating context.
OAuth2 PKCE
A secure authorization flow for apps that can't keep secrets (like mobile apps). The Whoop integration uses a local HTTP server on port 8080 to capture the authorization code, then exchanges it for access + refresh tokens.
SSE (Server-Sent Events)
A protocol where the server streams data to the browser over a single HTTP connection. Used by POST /query/stream to send Claude's answer token-by-token, so you see words appearing in real time instead of waiting for the full answer.
MCP (Model Context Protocol)
A JSON-RPC 2.0 protocol over stdin/stdout that lets AI assistants like Claude Desktop call external tools. The mcp_server.py implements this so Claude Desktop can query your life-intelligence graph directly as "tools".
Prompt Caching
An Anthropic API feature (beta) that caches the system prompt on Anthropic's servers between calls. When the same system block is sent again within the TTL, Anthropic skips re-processing it — saving ~80% of system-prompt tokens per request.
Semantic Cache
An in-memory dictionary keyed by a SHA-256 hash of the question + domains. If the exact same question is asked again within the TTL, the cached answer is returned immediately without calling Claude. Disabled when conversation history is active.
Apple Health Export
An XML file exported from the iPhone Health app containing years of health data — steps, heart rate, sleep, workouts, glucose, weight, and more. The system parses it with stdlib only (xml.etree.ElementTree) and maps it to the same node types used by Whoop and other sources.
